## Findings register

### F1 — P0 — Injury Support dead-ends without collecting injury context — ✅ FIXED `c0ff0e9`
- **Route:** /mobility (Injury Support mode), also reached from Train.
- **Repro (owner):** "Start Injury Support" (Suggested-Today card AND the
  "Injury support needs one more step" library card) returns to the previous
  screen; the user is never asked what hurts. Both buttons loop.
- **User impact:** the flagship safety feature collects no context; a user
  with a knee injury gets no path to knee-appropriate support. Unsafe +
  trust-destroying.
- **Root cause:** both CTAs were `EmptyStateCard ctaTo="/injury-support"` —
  an interstitial whose only actions are "Open Injury Support" (→ /mobility)
  and "Go to Dashboard": a closed loop. The real chooser (issue type → body
  area → results) existed but was buried inside the collapsed "Show filters"
  panel where no user would find it.
- **Fix:** inline `InjurySupportChooser` where the CTA was — Step 1 Injury /
  Ache, Step 2 specific injury (10 mappings) or body area (8) + symptom, plus
  a "From your profile" one-tap shortcut when onboarding declared restricted
  areas. Summary + Change once results show. The no-match state switches to
  Mobility & Stretch in place (third loop CTA removed).
- **Verified (browser, as-user):** ACL support → heel slide/quad set/short-arc
  quad/Spanish squat (knee rehab); Tennis elbow → forearm/grip/wrist work;
  Lower back · Stiffness → thoracic/hip work. Results change per answer.
  qa:launch 19/19.

### F2 — P0 — Workout category filters return identical exercises — ✅ FIXED `4fd83ab`
- **Route:** /workouts (category cards: Advanced / Joint-Friendly / 40+ / etc.)
- **Repro (owner + reproduced):** Joint-Friendly and 40+ returned
  byte-identical workout lists; nothing joint-adapted about them.
- **User impact:** unsafe recommendations presented under a safety label.
- **Root causes (three compounding):**
  1. The server tagged "40+" from the USER's ageGroup (everything for a 40+
     user, nothing for everyone else) — and its slug ("40plus") never matched
     the taxonomy id ("forty_plus"), so the category could never match at all.
  2. Zero strict matches silently fell back to the unfiltered list — including
     under safety labels (the two identical lists were both the fallback).
  3. A category with no focus mapping inherited the PREVIOUS category's focus
     (e.g. Recovery Day pinned mobility_recovery), starving the pool so every
     later selection looked the same.
- **Fix:** Joint-Friendly = sessions containing no high-joint-stress movement;
  40+ = additionally not high intensity (workout-derived, never user-derived);
  slug alias for forty_plus; category selection always resolves focus (mapped
  or recommended); safety filters (joint-friendly/40+/recovery-day/low joint
  stress) NEVER silently substitute — zero matches shows an honest empty state
  routing to guided mobility.
- **Verified (browser + API ground truth):** Advanced = the 2 dense leg
  sessions; 40+ = the 5 moderate sessions (Advanced ∩ 40+ = ∅); Recovery Day
  refuses with the safety note. qa:launch 19/19.
- **Queued follow-up (P1):** goal tags (Fat Loss / Muscle Building / Strength)
  are still added from the user's goal → those categories are too broad; the
  per-setup candidate pool is only ~7 sessions so categories overlap heavily —
  generator should synthesize category-specific variants.

### F5 — P1 — Exercise Library equipment filter returns zero for most selections — ✅ FIXED
- **Route:** /exercise-library (Equipment select).
- **Repro:** Chest (17 exercises) + "Barbell or dumbbell" → 0 matches, even
  though barbell/dumbbell bench presses exist.
- **Root cause:** the filter options were the catalog's RAW free-text
  equipment strings ("Barbell or dumbbell", "Barbell or dumbbells",
  "Barbell, dumbbells, or kettlebells depending on variation"…) matched by
  exact string equality — most selections could never match an entry.
- **Fix:** canonical equipment families (Barbell / Dumbbells / Kettlebells /
  Bands / Machines-Cables / Bench / Pull-up bar / Bodyweight) with token
  matching over the free text; options only offered when ≥1 entry matches;
  ?equipment= deep links normalized.
- **Verified (browser):** Dumbbells → 55 · Barbell → 17 · Machines/Cables →
  37 · Chest+Dumbbells → 5 (all real dumbbell chest movements).

### F7 — P1 — Coach "Action" cards were completely inert — ✅ FIXED
- **Route:** /coach → "Do these next".
- **Repro:** the ACTION card ("Start a workout") is a plain `<article>` — no
  click behavior at all. The coach's primary output couldn't be acted on.
- **Root cause:** the server's next-action builder HAS the route
  (`nextAction.to`) but dropped it from the payload; the client rendered
  static articles.
- **Fix:** payload carries `to` (primary + secondary insights); cards with a
  route render as links with a visible "Go →" cue, hover/focus states.
- **Verified (browser):** card renders as `<a href="/workouts">`, click lands
  on /workouts. qa:launch 19/19.
